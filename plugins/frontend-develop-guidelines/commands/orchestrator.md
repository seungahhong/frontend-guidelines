# Orchestrator — 프론트엔드 개발 전체 워크플로우 오케스트레이터

사용자의 요구사항을 받아 **분석 → PRD 작성 → 가이드라인 기반 개발**까지 전체 흐름을 서브에이전트로 순차 실행하고, 최종 결과를 통합하여 전달하는 커맨드.

## 왜 오케스트레이터가 필요한가

프론트엔드 개발은 요구사항 분석, 설계, 구현이 긴밀하게 연결된다. 각 단계를 독립된 서브에이전트로 분리하면 단계별 전문성을 살리면서도 컨텍스트 오염 없이 깊이 있는 작업이 가능하다. 오케스트레이터는 이 서브에이전트들 사이에서 산출물을 전달하고 흐름을 제어하는 지휘자 역할을 한다.

## 전체 파이프라인

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  사용자 요구사항                                              │
│       │                                                      │
│       ▼                                                      │
│  ┌─────────────┐                                             │
│  │  1. Analyst  │  요구사항 분석 · grill-me 인터뷰            │
│  │  (서브에이전트) │  → 요구사항 명세서                         │
│  └──────┬──────┘                                             │
│         │ 요구사항 명세서 전달                                  │
│         ▼                                                    │
│  ┌─────────────┐                                             │
│  │   2. PRD    │  계획 → 설계 → 비평 루프                     │
│  │  (서브에이전트) │  → 개발 요구사항 정의서                     │
│  └──────┬──────┘                                             │
│         │ PRD 전달                                            │
│         ▼                                                    │
│  ┌──────────────────┐                                        │
│  │ 3. Frontend      │  a11y · semantic-html · seo-geo · tdd  │
│  │    Guidelines    │  → 가이드라인 기반 개발                   │
│  │  (서브에이전트)    │  → 최종 점검 결과                        │
│  └──────┬───────────┘                                        │
│         │                                                    │
│         ▼                                                    │
│  ┌──────────────────┐                                        │
│  │  4. 최종 통합     │  오케스트레이터가 전 단계 결과를 통합     │
│  │   리포트 생성     │  → 사용자에게 전달                       │
│  └──────────────────┘                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 실행 절차

### Phase 0: 요구사항 접수

사용자에게 개발하려는 기능이나 프로젝트에 대해 설명을 요청한다. 이미 대화 중 요구사항이 제시되었다면 이 단계를 건너뛴다.

```
프론트엔드 개발 전체 워크플로우를 시작합니다.

분석 → PRD 작성 → 가이드라인 기반 개발 순으로 진행됩니다.
각 단계는 독립 서브에이전트가 수행하고, 저는 결과를 통합하여 전달합니다.

개발하려는 기능이나 프로젝트에 대해 설명해주세요.
```

사용자의 요구사항 원문을 그대로 보존한다. 이후 각 서브에이전트에 전달할 때 원문을 포함해야 맥락이 유지된다.

### Phase 1: Analyst 서브에이전트 — 요구사항 분석

Agent 도구로 **analyst 서브에이전트**(subagent_type: `frontend-develop-guidelines:analyst`)를 spawn한다.

**프롬프트 구성:**
```
1. 사용자의 요구사항 원문
2. 에이전트 정의: ./agents/analyst.md 의 프로세스를 따를 것
3. 코드베이스 경로 및 관련 컨텍스트
4. 산출물 요청: 요구사항 분석 결과 (기능/비기능 요구사항, 기술적 제약, 미해결 사항, 결정 로그)
```

analyst 에이전트는 내부적으로 grill-me 스킬을 활용하여 사용자에게 질문하고, 코드베이스를 조사하여 요구사항 명세를 도출한다.

**수집할 산출물:**
- 요구사항 분석 결과 (기능/비기능 요구사항 표, 기술적 제약, 결정 로그)

서브에이전트 완료 후, 산출물을 사용자에게 요약하여 보여주고 다음 단계 진행을 안내한다:

```
[Analyst 완료] 요구사항 분석이 완료되었습니다.

- 기능 요구사항: N건
- 비기능 요구사항: N건
- 기술적 제약: N건

다음 단계(PRD 작성)로 진행합니다.
```

### Phase 2: PRD 서브에이전트 — 개발 요구사항 정의서 작성

Agent 도구로 **PRD 서브에이전트**(subagent_type: `frontend-develop-guidelines:prd`)를 spawn한다.

**프롬프트 구성:**
```
1. 사용자의 요구사항 원문
2. Phase 1에서 수집한 요구사항 명세 전문
3. 에이전트 정의: ./agents/prd.md 의 프로세스를 따를 것
4. 코드베이스 경로 및 관련 컨텍스트
5. 산출물 요청: 확정된 PRD (계획 → 설계 → 비평 루프를 거친 최종본)
```

PRD 에이전트는 내부적으로 planner, architecture, critic 각각을 서브에이전트로 spawn하여 루프를 돌린다. 오케스트레이터는 PRD 에이전트의 최종 산출물만 수집한다.

**수집할 산출물:**
- 확정된 PRD (구현 계획, 기술 아키텍처, 리스크 대응, 인수 조건 포함)

서브에이전트 완료 후, PRD 핵심 내용을 사용자에게 요약하여 보여준다:

```
[PRD 완료] 개발 요구사항 정의서가 확정되었습니다.

- 구현 단계: N개 Phase
- 수정 대상 파일: N개
- 리스크: 상 N건 / 중 N건 / 하 N건
- PRD 저장 위치: .claude/plans/{기능명}-prd.md

다음 단계(가이드라인 기반 개발)로 진행합니다.
```

### Phase 3: Frontend Guidelines 서브에이전트 — 가이드라인 기반 개발

Agent 도구로 **frontend-guidelines 서브에이전트**(subagent_type: `frontend-develop-guidelines:frontend-guidelines`)를 spawn한다.

**프롬프트 구성:**
```
1. 사용자의 요구사항 원문
2. Phase 1의 요구사항 명세 전문
3. Phase 2의 확정된 PRD 전문
4. 에이전트 정의: ./agents/frontend-guidelines.md 의 프로세스를 따를 것
5. 코드베이스 경로 및 관련 컨텍스트
6. 산출물 요청: 가이드라인 적용 결과 (코드 수정 + 최종 점검 결과)
```

frontend-guidelines 에이전트는 내부적으로 선택된 스킬(a11y, semantic-html, seo-geo, tdd) 각각을 서브에이전트로 spawn하여 병렬 실행한 뒤 통합한다. 오케스트레이터는 최종 통합 결과만 수집한다.

**수집할 산출물:**
- 적용된 가이드라인 목록 및 각 스킬별 점검 결과
- 코드 수정 내역
- 최종 점검 결과표

### Phase 4: 최종 통합 리포트

모든 서브에이전트의 산출물을 수집하여 하나의 통합 리포트를 작성한다.

```markdown
# [프로젝트/기능명] 개발 완료 리포트

## 1. 워크플로우 요약
| 단계 | 에이전트 | 상태 | 주요 산출물 |
|------|----------|------|------------|
| 요구사항 분석 | Analyst | ✅ | 요구사항 명세서 |
| PRD 작성 | PRD | ✅ | 개발 요구사항 정의서 |
| 가이드라인 개발 | Frontend Guidelines | ✅ | 코드 수정 + 점검 결과 |

## 2. 요구사항 분석 요약
[Phase 1 핵심 내용]

## 3. PRD 핵심 내용
[Phase 2 핵심 내용 — 구현 계획, 아키텍처, 리스크]

## 4. 가이드라인 적용 결과
### 적용된 가이드라인
| 스킬 | 상태 | 비고 |
|------|------|------|
| ... | ✅/⚠️/❌ | ... |

### 수정된 파일
| 파일 | 변경 유형 | 설명 |
|------|----------|------|

## 5. 미해결 사항 및 후속 작업
- [ ] ...

## 6. 결정 로그
| 항목 | 결정 | 근거 | 단계 |
|------|------|------|------|
```

## 단계 간 산출물 전달 규칙

각 서브에이전트는 독립된 컨텍스트에서 실행되므로, 이전 단계의 산출물을 프롬프트에 **전문 포함**하여 전달해야 한다. 요약이나 발췌가 아닌 전문을 전달하는 이유는, 서브에이전트가 이전 맥락을 직접 본 적이 없기 때문이다.

- Phase 2 (PRD) → Phase 1의 요구사항 명세 전문을 프롬프트에 포함
- Phase 3 (Frontend Guidelines) → Phase 1의 명세 + Phase 2의 PRD 전문을 프롬프트에 포함

산출물이 지나치게 긴 경우(토큰 제한 우려), 핵심 섹션을 선별하되 "기능 요구사항 표", "구현 계획", "수정 대상 파일" 등 구현에 직접 필요한 정보는 반드시 포함한다.

## 사용자 소통

- 각 Phase 시작 시: "N단계(에이전트명)를 시작합니다" 안내
- 각 Phase 완료 시: 핵심 산출물 요약 + 다음 단계 안내
- 서브에이전트가 사용자 입력을 필요로 하는 경우(analyst의 인터뷰, frontend-guidelines의 스킬 선택): 해당 질문을 사용자에게 전달하고 응답을 서브에이전트에 전달한다
- 문제 발생 시: 어떤 단계에서 무엇이 문제인지 명확히 전달하고, 재시도 여부를 사용자에게 확인한다

## 조기 종료

사용자가 특정 단계까지만 진행하길 원할 수 있다. 예를 들어 "분석과 PRD까지만 해줘"라고 하면 Phase 2까지만 실행하고 통합 리포트를 생성한다. 통합 리포트에는 실행된 단계만 포함한다.
